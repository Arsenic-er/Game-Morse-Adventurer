# CW 台站模拟游戏设计文档

**文档版本：** v0.4（对话整理合并版）  
**项目状态：** 概念收敛 / MVP 设计阶段  
**目标平台：** Windows / Steam  
**开发方式：** 单人开发，程序与美术主要借助 AI  
**发布方向：** 开源，并面向 Ham 社区公开  

---

## 1. 项目概述

本项目是一款面向业余无线电爱好者的 **2D 像素风、单机 CW 台站模拟与练习游戏**。

目标玩家主要是：

- 已经了解业余无线电；
- 对 CW 有兴趣；
- 但尚未熟练掌握 CW 收发、呼号抄收和基本 QSO 流程的人。

游戏不采用传统 RPG 的战斗、角色等级、生命值或装备稀有度体系，也暂时不设置故事主线。核心体验来自玩家自身技术的成长：

- 从听不懂 CW 到能够辨认字符；
- 从不会发报到能够稳定使用直键或双桨；
- 从只能完成近距离、强信号通联，到能够使用更好的设备和天线联系更远地区；
- 从依赖视觉提示，到可以仅凭声音完成基础 QSO。

项目的核心理念是：

> 玩家升级的不只是游戏里的设备，更是玩家本人真实的 CW 能力。

---

## 2. 产品定位

### 2.1 产品形态

- 单机游戏；
- Windows 优先；
- 计划发布于 Steam；
- 开源并在 Ham 社区公开；
- 无真人在线通联；
- 联网只用于未来读取公开传播环境数据，不属于联机玩法；
- 断网时仍可完整运行。

### 2.2 拟真程度

拟真程度暂定为 **3 / 5**：

- 保留真实 CW 的基本节奏、流程、设备逻辑和传播概念；
- 使用可信、容易理解的简化模型；
- 不追求工程级电离层、天线、射频和接收机仿真；
- 不让复杂技术参数阻碍玩家开始练习。

### 2.3 美术与交互方向

- 2D 像素画；
- 非 RPG；
- 主要交互集中在电台设备、练习台、传播地图、日志和设置界面；
- 设备旋钮、指示灯、仪表、电键动作需要有明确视觉反馈；
- 暂不制作角色在多个房间中自由行走的系统；
- 暂不制作长篇剧情和分支对话。

## 4. 核心设计原则

### 4.1 玩家能力不可购买

游戏设备可以改善通联条件，但不能替玩家完成听抄和发报。

例如：

- 更好的天线可以让原本很弱的地区变得可通联；
- 更高功率可以增加远距离回应的机会；
- 未来的瀑布图可以帮助发现频段活动；
- 但任何设备都不能直接自动填写 NPC 呼号或替玩家发送完整 QSO。

### 4.2 设备升级必须改变体验

设备升级不能只表现为后台数值增加，而应在以下方面产生可感知变化：

- 地图上更多地区由“不可用”变为“可通联”；
- NPC 信号更容易出现；
- 信号相对背景噪声更清楚；
- 能够使用新的频率或频段；
- 能够获得瀑布图、调谐、滤波等新的操作能力。

### 4.3 传播系统要直观

玩家不需要先理解复杂电离层术语。游戏只向玩家展示：

- 当前频率；
- 当前 UTC 和当地时间；
- 世界各地区在 21.060 MHz 上的大致传播等级；
- 自己当前设备能否完成通联。

### 4.4 训练结果应尽量迁移到真实 CW

需要避免玩家只适应一种固定声音：

- 音调可调；
- 速度可调；
- NPC 可有轻微频偏；
- 信号可有背景噪声和 QSB；
- 视觉提示可关闭；
- 练习内容使用随机字符和虚构呼号，不长期重复固定答案。

## 5. MVP 定义

### 5.1 MVP 一句话定义

> 玩家在世界地图上选择虚构台站位置，获得固定工作于 21.060 MHz 的低功率 `squid01` 套件；先在练习台调整音调和速度，练习接收、直键和双桨发报，然后在当前现实时间对应的简化传播环境中，与无发报错误的 NPC 交换呼号、RST 和 73。完成 QSO 后获得信用点并写入日志。

### 5.2 MVP 必须包含

- 开始界面；
- 虚构呼号免责声明；
- 世界地图选点；
- 玩家位置、UTC 和当地时间显示；
- 电台主界面；
- `squid01` 面板；
- CW 声音生成；
- 音调与速度设置；
- 字符接收练习；
- 呼号接收练习；
- 直键发报练习；
- 双桨发报练习；
- 可开关的视觉辅助；
- NPC 呼叫 CQ；
- 最小 QSO 状态流程；
- 默认 21 MHz 全球传播地图；
- 背景噪声、QSB 和轻微频偏；
- 大功率强台的低概率出现；
- QSO 结果页；
- 信用点；
- 基础日志。

### 5.3 MVP 暂不包含

- 故事主线；
- 多频段；
- 连续调谐；
- 瀑布图；
- 复杂滤波器；
- 商店和完整设备树；
- 真人在线通联；
- 真实电台控制；
- 外接真实电键；
- 复杂电离层物理模型；
- 邻频 QRM、同频多台、雷电 QRN、工业噪声等独立系统；
- NPC 发报错误；
- NAME、QTH、天气、设备介绍和长文本交流；
- Pileup、竞赛序号和复杂重发流程。

## 6. 核心游戏循环

```text
进入游戏
→ 查看当前时间和传播地图
→ 进入练习台或电台
→ 调整音调和速度
→ 听取 NPC 的 CQ
→ 抄收 NPC 呼号
→ 使用直键或双桨回应
→ 交换 RST
→ 发送 73
→ 完成 QSO
→ 获得信用点
→ 写入日志并更新最远通联记录
→ 后续使用信用点换取更好的设备或天线
```

MVP 阶段尚未实现完整设备购买，但需要从第一版开始记录信用点，为后续设备成长保留接口。

## 7. 世界地图、地点与时间

### 7.1 初始地点选择

新游戏流程：

1. 显示虚构呼号和频率免责声明；
2. 打开世界地图；
3. 玩家使用鼠标点击任意地点；
4. 系统保存该地点经纬度；
5. 系统根据地点自动确定当地时区；
6. 系统生成玩家的虚构呼号；
7. 玩家确认并进入电台。

### 7.2 时间规则

游戏内部统一使用 UTC。

界面同时显示：

```text
UTC：12:00
LOCAL：根据玩家地点自动换算
```

规则：

- 全世界在同一时刻的 UTC 相同；
- 玩家地点只影响当地时间显示；
- 当前版本时间与现实时间同步；
- 暂不支持快进、睡觉、跳过白天或选择历史日期；
- 玩家重新进入游戏时读取新的现实时间；
- 传播地图依据当前 UTC、玩家地点和 21.060 MHz 生成。

### 7.3 时区实现要求

- 玩家不需要手动输入 UTC 偏移；
- 系统根据地图点击位置自动确定；
- 存档保存经纬度、时区标识和玩家地点名称；
- 夏令时问题由时区数据处理，不由玩家手动配置。

## 8. 21 MHz 传播地图

### 8.1 目标

传播地图只需要回答：

> 从玩家当前地点出发，在当前 UTC 和 21.060 MHz 条件下，世界各地区大致有多容易完成通联？

游戏内统一称为：

> **21 MHz 传播地图**

不向玩家展示复杂的电离层指标，也不要求玩家理解专业传播图。

### 8.2 重要影响因素

MVP 只保留几个最重要、容易建立关系的因素：

#### 当前频率

- MVP 固定为 21.060 MHz；
- 因此不需要在多个频率之间比较；
- 后续多频设备才加入其他频率或频段。

#### 当前 UTC 和昼夜分布

- 时间变化会使世界的日照区、夜间区和晨昏区域移动；
- 传播地图随现实 UTC 更新；
- 当前阶段不让玩家跳过时间。

#### 玩家地点与 NPC 地点

- 玩家地点是传播地图的起点；
- NPC 所在地决定读取地图上的哪个区域；
- 不同玩家初始地点看到的传播分布不同。

#### 玩家设备和 NPC 台站能力

- 地图给出基础传播等级；
- 玩家设备和天线可以提供简单加成；
- 大功率强台可以获得额外 NPC 加成。

### 8.3 五级传播结果

| 等级 | 名称 | 游戏表现 |
|---:|---|---|
| 0 | 不可用 | 不生成普通可完成 QSO 的 NPC |
| 1 | 很弱 | 普通台通常不可通，偶尔出现大功率强台 |
| 2 | 可通联 | 信号较弱，背景噪声和 QSB 较明显 |
| 3 | 良好 | 普通 NPC 可以较稳定通联 |
| 4 | 很强 | 信号较清楚，仅有轻微噪声和衰落 |

地图使用五种清楚的像素色块，不显示虚假的精确小数。

### 8.4 简化关系

```text
最终传播等级 = 地图基础等级 + 玩家设备加成 + NPC 台站加成
```

结果限制在 0 至 4：

```text
FinalLevel = clamp(
    BaseLevel
    + PlayerEquipmentBonus
    + NpcStationBonus,
    0,
    4
)
```

MVP 中：

```text
squid01 设备加成：0
基础天线加成：0
普通 NPC 加成：0
强台 NPC 加成：+1
极少数特殊强台：可暂留 +2 的配置能力
```

### 8.5 地图分辨率

推荐第一版使用低分辨率网格：

```text
72 × 36 格
每格约代表 5° 经度 × 5° 纬度
```

优点：

- 易于像素化显示；
- 数据量小；
- 易于缓存；
- 易于测试；
- 玩家可以快速理解世界传播分布。

### 8.6 在线与断网规则

#### MVP

第一版不依赖任何外部网站，使用内置默认传播环境。

#### 后续联网增强

未来有网络时：

```text
读取可信的公开传播结果
→ 转换成游戏的 0—4 级世界地图
→ 供 NPC 和音频系统使用
```

断网或读取失败时：

```text
直接使用默认传播环境
```

不弹出阻断式错误，不影响练习、QSO、信用点或存档。

所有来源最终统一输出相同格式：

```text
PropagationMap
- frequency_mhz
- generated_at_utc
- source
- grid_width
- grid_height
- cells[]
```

来源可标记为：

```text
OFFLINE_DEFAULT
ONLINE_OFFICIAL
CACHED_ONLINE
```

### 8.7 默认传播环境

默认环境根据以下内容生成：

- 玩家位置；
- NPC 位置；
- 当前 UTC；
- 固定频率 21.060 MHz。

初版可以使用按 UTC 时段划分的简化模板：

| UTC 时段 | 使用模板 |
|---|---|
| 00:00—05:59 | 模板 A |
| 06:00—11:59 | 模板 B |
| 12:00—17:59 | 模板 C |
| 18:00—23:59 | 模板 D |

模板以玩家地点为起点进行旋转或映射。相同地点、相同 UTC 时段的整体结果应大致稳定，避免传播地图每次打开都完全随机。

## 9. 接收音频效果

### 9.1 保留的效果

传播系统只生成最终听感，不分别向玩家解释噪声来源。

MVP 保留：

- 通用背景噪声；
- QSB 信号衰落；
- NPC 轻微频偏。

### 9.2 暂不单独模拟

- 邻频 QRM；
- 同频多个台站；
- 雷电 QRN；
- 工业和家用电器噪声；
- 脉冲干扰；
- 恶意干扰；
- 单独的太阳爆发或磁暴音效系统。

这些效果未来如需加入，也应先统一成最终接收结果，而不是一次性做成复杂物理模拟。

### 9.3 与传播等级的关系

| 最终等级 | 信号表现 |
|---:|---|
| 0 | 不生成正常可完成的 QSO |
| 1 | 信号很弱，只有强台可能出现；底噪高、QSB 深 |
| 2 | 信号可抄但较困难；底噪明显、QSB 中等 |
| 3 | 信号清楚；有轻微噪声和衰落 |
| 4 | 信号较强；仅有轻微背景噪声和缓慢起伏 |

传播系统只改变声音，不直接随机决定玩家是否抄错。

## 10. NPC 系统

### 10.1 NPC 类型

MVP 先实现三类：

| 类型 | 特征 |
|---|---|
| 练习台 | 慢速、标准、无错误、信号稳定 |
| 普通台 | 正常速度、轻微频偏、无发报错误 |
| 强力 DX 台 | 距离较远，拥有较高 NPC 台站加成，偶尔在初期出现 |

### 10.2 NPC 参数

```text
NpcStation
- callsign
- region_id
- latitude
- longitude
- wpm
- base_tone_hz
- frequency_offset_hz
- station_bonus
- is_strong_station
- error_rate
- is_fictional
```

MVP 中：

```text
error_rate = 0
is_fictional = true
```

### 10.3 大功率强台

为了让初始低功率玩家偶尔体验远距离通联，可以低概率生成强台。

推荐初始配置：

```text
普通 NPC：约 90%
强台 NPC：约 10%
```

实际概率后续通过试玩调整。

强台加成代表对方可能同时拥有：

- 更高发射功率；
- 更好的天线；
- 更好的接收条件。

MVP 不拆分上行和下行链路。

### 10.4 NPC 发报错误

MVP 所有 NPC 发报错误率为 0。

原因：

- 初学者需要先建立对标准 CW 和标准 QSO 的信任；
- 玩家失败时应能判断是自己没有听清或没有按流程回应；
- 不应让初学者怀疑系统或 NPC 是否发错。

后期高级模式可加入极低概率的人为错误，并默认关闭。

## 11. 虚构呼号系统

### 11.1 开始界面免责声明

建议显示：

> 本游戏中的所有呼号、台站、人物和通联记录均为程序生成的虚构内容，与现实中的任何个人、组织或无线电台无关。任何相似均属巧合。本游戏中的频率和操作内容不构成现实无线电操作建议。

玩家确认后才能进入地图选点。

### 11.2 生成规则

NPC 呼号根据所在地使用对应的地区呼号前缀，再随机增加一至两个英文字母。

```text
NPC 呼号 = 地区呼号前缀 + 随机 1—2 个英文字母
```

例如只定义结构，不在文档中固定真实示例：

```text
[地区前缀] + A
[地区前缀] + QX
```

### 11.3 风险控制

- NPC 数据统一标记 `is_fictional = true`；
- 日志和结果页可显示小型 `SIM` 标记；
- 游戏不从真实呼号数据库抓取个人资料；
- 不生成现实姓名、地址或私人信息；
- 后续可维护虚构呼号排除表；
- 免责声明必须保留。

仅增加一两个字母不能绝对保证不与现实呼号相同，因此需要通过免责声明、SIM 标记和后续排除机制共同降低风险。

## 12. 最小 QSO 流程

### 12.1 目标

第一个可玩版本只验证：

- 玩家能否听懂 NPC 呼号；
- 玩家能否正确发送双方呼号；
- 玩家能否发送一个有效 RST；
- 玩家能否发送 73；
- 系统能否判定并记录一次完整基础 QSO。

### 12.2 QSO 状态

```text
WAITING_CQ
→ PLAYER_REPLY
→ NPC_RST
→ PLAYER_RST_AND_73
→ NPC_73_AND_SK
→ QSO_COMPLETE
```

### 12.3 示例流程

NPC：

```text
CQ CQ DE [NPC呼号] K
```

玩家：

```text
[NPC呼号] DE [玩家呼号] K
```

NPC：

```text
[玩家呼号] DE [NPC呼号] RST 579 K
```

玩家：

```text
[NPC呼号] DE [玩家呼号] RST 559 73 K
```

NPC：

```text
R 73 SK
```

### 12.4 成功条件

- 正确抄收 NPC 呼号；
- 正确发送 NPC 呼号；
- 正确发送玩家呼号；
- 发送一个格式有效的 RST；

- QTH；

## 14. 发报输入、音调与速度

### 14.1 输入方式

#### 直键

默认映射可为：

- 键盘空格键；
- 鼠标左键；
- 玩家自定义按键。

系统记录：

```text
KEY_DOWN
KEY_UP
```

根据按下时长判断点或划。

#### 双桨

默认映射可为：

| 动作 | 键盘 | 鼠标 |
|---|---|---|
| 点桨 | Z 或左方向键 | 左键 |
| 划桨 | X 或右方向键 | 右键 |
| 双桨同时按下 | 同时按两键 | 同时按左右键 |

所有按键都应允许重新绑定。

### 14.2 音调

推荐范围：

```text
400—1000 Hz
```

音调影响：

- 玩家听到的发报侧音；
- 练习台接收音；
- NPC 基础音调与轻微频偏后的结果。

音调不直接影响字符判定。

### 14.3 速度

推荐范围：

```text
5—30 WPM
```

#### 直键

- 玩家自己控制按下和松开的时间；
- 系统根据当前目标速度识别点、划和间隔；
- 判定应有合理容差，不要求机器级精度。

#### 双桨

- 玩家输入点桨或划桨；
- 系统根据 WPM 自动生成标准时值；
- 玩家仍需决定正确字符和 QSO 内容。

### 14.4 输入技术要求

- 不依赖画面帧率计算按键时长；
- 使用高精度、单调递增的时间戳；
- 不使用操作系统键盘自动重复作为新输入；
- 输入事件能够保存和回放；
- 鼠标和键盘都需要独立校准与测试；
- 底层预留未来真实电键输入适配器。

建议结构：

```text
键盘输入 ─┐
鼠标输入 ─┼→ KeyInputSource → KeyerEngine → SidetoneEngine
真实电键 ─┘                       ↓
                               TimingAnalyzer
                                      ↓
                              Decoder / Scorer
```

真实电键适配器当前不实现。

## 23. 未知中的未知与风险控制

### 23.1 玩家只适应游戏声音

风险：

- 只会听固定音调；
- 只会听完美节奏；
- 关闭视觉辅助后表现大幅下降。

控制：

- 音调可调；
- NPC 有轻微频偏；
- 练习内容随机；
- 无辅助模式；
- 后期增加更多节奏和信号条件。

### 23.2 鼠标和键盘延迟影响评分

风险：

- 不同帧率得到不同结果；
- 鼠标声音与操作不同步；
- 双桨同时输入冲突。

控制：

- 高精度时间戳；
- 帧率无关判定；
- 可回放原始输入事件；
- 输入测试页面；
- 所有按键可重映射。

### 23.3 现实时间导致玩家当下难以通联

风险：

- 玩家只能在某些现实时间获得好传播；
- 游戏内容受现实作息限制。

当前决定：

- MVP 暂不允许跳过时间；
- 练习台始终可用；
- 默认传播环境应保证至少存在可完成 QSO 的区域；
- 通过偶发强台避免完全没有远方信号；
- 若测试证明现实时间严重影响可玩性，再考虑实验时间模式。

### 23.4 虚构呼号与真人碰撞

控制：

- 地区前缀加额外随机字母；
- SIM 标记；
- 免责声明；
- 不抓取现实个人资料；
- 后期排除表和社区审校。

### 23.5 开源项目中的 AI 素材来源

从第一天维护素材来源表：

```text
文件名
生成工具或来源
生成日期
提示词概要
参考素材来源
人工修改内容
许可证或使用条款
是否允许重新分发
```

避免后期无法判断图片、字体、声音或代码能否随开源项目和 Steam 版本发布。

### 23.6 游戏成长退化为刷信用点

控制：

- 重复奖励递减；
- 新地区和新距离提供额外奖励；
- 后续设备可同时要求信用点和技能里程碑；
- 设备不能自动解码。

- 玩家可以使用键盘或鼠标操作双桨；
- 字符接收和呼号接收练习可用；
- 视觉辅助可以开启或关闭；
- NPC 可以无错误地发送最小 QSO；
- 玩家可以正确回应呼号、RST 和 73；
- 背景噪声、QSB 和轻微频偏会随传播等级变化；
- 默认传播地图能生成可通联 NPC；
- 强台偶尔能让初始设备完成较远 QSO；
- 完成 QSO 后获得信用点并写入日志；
- 相同输入事件可以稳定得到相同识别结果；
- 画面帧率变化不会明显改变直键判定。

## 26. 当前最终方案摘要

本项目当前确定为一款开源的 Windows 单机 2D 像素 CW 台站模拟游戏。玩家在世界地图上选择台站地点，系统自动显示 UTC 和当地时间，并生成虚构呼号。玩家从固定 21.060 MHz、低功率、单频的 `squid01` 套件开始，通过练习台学习字符接收、呼号抄收、直键和双桨发报。

MVP 中，玩家与无发报错误的虚构 NPC 完成最小 QSO：交换呼号、RST 和 73。游戏使用一张面向玩家地点生成的五级 21 MHz 世界传播地图，并通过背景噪声、QSB 和轻微频偏表现传播效果。初始设备通常只能联系传播较好的地区，但偶尔会遇到大功率强台。通联成功后获得信用点并记录到日志。

第一阶段只开发 UI、练习台、`squid01`、默认传播地图和最小模拟 QSO。设备价格、多频段、瀑布图、复杂设备升级、真实电键和在线官方传播数据均放到后续阶段。
